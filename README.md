# Octocron - Distributed Cron Scheduler
# 🐙 Octocron

**Отказоустойчивый распределённый планировщик задач с удалённым выполнением по SSH**

Octocron — это кластерный cron-сервер, написанный на Go. Он позволяет централизованно управлять регламентными операциями (перезапуск служб, бэкапы и т.д.) на десятках и сотнях серверов. Благодаря встроенному Raft-кластеру, система не имеет единой точки отказа и продолжает работу даже при выходе из строя управляющих узлов.

![Архитектура](https://img.shields.io/badge/Architecture-Raft%20%2B%20Stateless%20Cron-blue)

## ✨ Ключевые особенности

- **Отказоустойчивость** (HA-кластер на Raft из 3+ узлов).
- **Stateless cron**: задания не дублируются при смене лидера.
- **Удалённое выполнение**: команды запускаются на целевых хостах по SSH.
- **История выполнения**: сохраняется stdout, stderr и код возврата для каждой задачи.
- **Удобный CLI-клиент** (`octoctl`) с автодополнением.
- **Единый бинарник**: минимальные зависимости, легко развернуть в закрытом контуре.

## 🏛 Архитектура

1. **Raft-кластер** хранит состояние (задания, целевые хосты, историю) и обеспечивает консенсус.
2. **Stateless cron** на лидере раз в секунду проверяет расписание и выполняет команды на удалённых машинах.
3. **gRPC API** для управления заданиями и хостами.
4. **CLI-клиент** для удобной работы из командной строки.

Все данные реплицируются между узлами. При падении лидера один из последователей автоматически становится новым лидером и продолжает выполнение задач без потерь и дублирования.

## 📦 Установка

### Из исходников
```bash
git clone https://github.com/NillHellberg/octocron.git
cd octocron
make build
sudo make install

После установки команды octocron-server и octoctl будут доступны глобально.

##Зависимости
Go 1.22+

Доступ к целевым хостам по SSH (приватный ключ должен находиться на всех узлах кластера)


🚀 Быстрый старт (тестовый кластер из 3 нод)
Предположим, у вас три сервера с IP-адресами 10.0.0.98 (octo1), 10.0.0.83 (octo2) и 10.0.0.91 (octo3). Бинарные файлы уже скопированы на все узлы (либо скомпилированы на месте).

На первом узле (octo1, 10.0.0.98):

bash
octocron-server --node-id=octo1 --bind-addr=10.0.0.98:12000 --grpc-addr=0.0.0.0:50051 --data-dir=./data
На втором узле (octo2, 10.0.0.83):

bash
octocron-server --node-id=octo2 --bind-addr=10.0.0.83:12000 --grpc-addr=0.0.0.0:50051 --data-dir=./data --join=10.0.0.98:50051
На третьем узле (octo3, 10.0.0.91):

bash
octocron-server --node-id=octo3 --bind-addr=10.0.0.91:12000 --grpc-addr=0.0.0.0:50051 --data-dir=./data --join=10.0.0.98:50051
Логи покажут Bootstrapped single-node cluster (octo1) и Successfully joined cluster (octo2, octo3). Лидером станет octo1.

Добавьте целевой хост (на любом узле кластера, запрос автоматически попадёт на лидера):

bash
octoctl --server=127.0.0.1:50051 target add --name target1 --address 10.0.0.175 --port 22 --user root --key /root/.ssh/octocron_target
Запомните ID из вывода (<TARGET_ID>).

Создайте задание:

bash
octoctl --server=127.0.0.1:50051 job create --name "log-writer" --cron "*/10 * * * * *" --command "echo 'Hello from Octocron' >> /tmp/octocron-demo.log" --target <TARGET_ID>
Проверьте выполнение:

bash
octoctl --server=127.0.0.1:50051 job list
octoctl --server=127.0.0.1:50051 job history <JOB_ID>
Каждые 10 секунд на целевой машине в /tmp/octocron-demo.log будет дописываться строка.

🧪 Тестирование отказоустойчивости
Остановите узел-лидер (Ctrl+C).

В логах одного из оставшихся узлов появится: I am leader, starting stateless cron, и задания продолжат выполняться без пропусков.

Запустите упавший узел снова с теми же параметрами (без --join). Он обнаружит существующее состояние (Existing cluster state found, skipping bootstrap) и присоединится как follower.

Важно: приватный ключ SSH должен лежать на всех узлах кластера, чтобы любой из них мог подключиться к целевым хостам.

📋 CLI-команды
Команда	Описание
octoctl target add	Добавить целевой хост
octoctl target list	Показать все хосты
octoctl target remove <ID>	Удалить хост
octoctl job create	Создать задание
octoctl job list	Показать все задания
octoctl job delete <ID>	Удалить задание
octoctl job history <ID>	История выполнения задания
Флаг --server (по умолчанию localhost:50051) задаёт адрес любого узла кластера. Запросы, изменяющие состояние (create, delete, add, remove), прозрачно перенаправляются на лидера.
